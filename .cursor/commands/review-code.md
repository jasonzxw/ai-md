# review-code

## 概述
针对本项目（Java Spring Boot + MyBatis + Redis 多模块架构）的代码审查指令。
审查时按 **P0 → P1 → P2 → P3** 优先级依次检查，每个发现标注严重等级标签。

## 严重等级定义

| 标签 | 含义 | 要求 |
|------|------|------|
| `[Must Fix]` | 阻断性问题，不修复不可合并 | 安全漏洞、数据损坏风险、逻辑错误 |
| `[Should Fix]` | 高风险问题，强烈建议本次修复 | 并发隐患、异常处理缺失、性能瓶颈 |
| `[Suggestion]` | 改进建议，可排入后续迭代 | 代码结构优化、可读性提升 |
| `[Nit]` | 极小问题，改不改均可 | 命名微调、格式细节 |

---

## P0：安全性审查（最高优先级）

### 身份认证与授权
- [ ] 每个对外接口是否都有鉴权逻辑？是否存在遗漏鉴权的新增接口？
- [ ] `AccountId.ACCOUNT_ID` 回退值是否仅用于开发环境？生产环境是否有可能绕过认证？

### SQL 注入防护
- [ ] MyBatis XML 中是否使用了 `${}` 拼接？如有，是否为**不可由用户控制**的值（如表名、列名常量）？
- [ ] 动态 SQL 的 `<if>` 条件拼接是否安全？是否存在条件全为空导致全表操作的风险？

### 敏感数据
- [ ] 接口返回值是否泄露了不应暴露的字段（如内部 ID、其他用户信息）？
- [ ] 日志中是否打印了 accountId 之外的敏感用户信息？
- [ ] 配置文件中是否有硬编码的密钥、Token 或密码？

---

## P1：正确性与健壮性审查

### 业务逻辑正确性
- [ ] 代码是否准确实现了需求描述的功能？
- [ ] 边界条件是否考虑完整（空值、零值、负值、超大值）？
- [ ] 业务状态流转是否正确？是否有不合法的状态跳转？

### 并发安全
- [ ] "先查询再操作"模式（check-then-act）是否存在竞态条件？是否需要分布式锁或数据库唯一约束？
- [ ] Redis 操作是否存在缓存击穿/穿透/雪崩风险？
- [ ] `@Transactional` 使用是否正确？事务范围是否过大或过小？是否存在事务内调用外部服务导致长事务的情况？

### 异常处理
- [ ] 是否有 catch 块吞掉异常却不记录日志？
- [ ] `Integer.parseInt()`、`Long.parseLong()` 等类型转换是否有 try-catch 或使用安全解析方式？
- [ ] 外部服务调用（HTTP、MQ）是否有超时设置和降级策略？
- [ ] Redis 操作失败时是否有合理的降级逻辑，而非直接抛异常导致主流程中断？

### 空指针防护
- [ ] DAO 查询结果可能为 null，调用方是否做了判空？
- [ ] 链式调用（如 `obj.getA().getB()`）中间环节是否可能为 null？
- [ ] 集合操作前是否检查了 null 和空集合？

---

## P2：性能与资源审查

### 数据库层
- [ ] 新增 SQL 是否会走索引？WHERE 条件字段是否有索引覆盖？
- [ ] 是否存在 N+1 查询（循环内执行 SQL）？
- [ ] 批量操作（IN 子句、批量插入）是否考虑了数据量上限？
- [ ] `<where>` 动态条件全为空时是否会导致全表扫描？

### 缓存层（Redis）
- [ ] 缓存 Key 命名是否遵循 `RedisConstKey` 统一规范？是否会与其他业务 Key 冲突？
- [ ] 缓存是否设置了合理的过期时间？是否存在永不过期的缓存？
- [ ] 数据变更时是否同步更新/删除了对应缓存？是否存在数据不一致窗口？

### 线程池与异步
- [ ] `@Async` 或 `CompletableFuture` 是否使用了项目中已定义的线程池，而非默认线程池？
- [ ] 异步任务中的异常是否被捕获和记录？
- [ ] 线程池任务是否有可能无限堆积导致 OOM？

### 接口性能
- [ ] 单个接口的调用链路中，数据库/Redis/外部服务的调用次数是否合理？
- [ ] 是否存在可以合并的多次查询？
- [ ] 大数据量查询是否有分页？

---

## P3：代码质量与可维护性审查

### 分层规范
- [ ] Controller 层是否只做参数校验和结果封装，不包含业务逻辑？
- [ ] Service 层是否只包含业务逻辑，不直接操作 `HttpServletRequest` 等 Web 层对象？
- [ ] DAO 层接口方法命名是否清晰（`selectXxxByXxx`、`insertXxx`、`updateXxxByXxx`）？

### 代码重复
- [ ] 是否存在在多个方法/类中重复的代码块？是否应该抽取为公共方法或工具类？
- [ ] Controller 中重复的鉴权、参数校验逻辑是否应提取到拦截器或注解中？

### 命名与可读性
- [ ] 类名、方法名、变量名是否准确表达其含义？
- [ ] 魔法数字是否提取为有意义的常量（参考已有的 `INTERVAL_DAY`、`POP_MODAL_CACHE_DAYS` 模式）？
- [ ] 复杂业务逻辑是否有必要的注释说明**意图和原因**（而非复述代码）？

### API 规范
- [ ] `@ApiOperation` 注解是否准确描述了接口功能？
- [ ] HTTP 方法是否语义正确（查询用 GET、修改用 POST/PUT）？
- [ ] `@RequestBody` 接收的对象是否有 `@Valid` 参数校验注解？
- [ ] 接口返回值是否统一使用 `CommonResp` 封装？`ErrorCode` 使用是否正确？

### MyBatis XML
- [ ] `resultMap` 字段映射是否与实体类属性完全一致？
- [ ] `<sql>` 片段是否被合理复用？
- [ ] 动态 SQL 的 `<if>` 条件中，第一个条件是否多写了 `AND`（`<where>` 标签可自动处理，但需确认）？

## 审查输出格式

对每个发现项，按以下格式输出：

```
### [严重等级] 问题简述
- **文件**：`文件路径:行号`
- **问题描述**：具体说明问题是什么
- **风险**：可能导致什么后果
- **建议修复方案**：给出具体的修改建议或代码示例
```

## 审查范围指引

根据审查场景灵活选择检查范围：

| 场景 | 建议范围 |
|------|---------|
| 新增接口 | 重点 P0 + P1 + P3（API 规范），快速过 P2 |
| 修复 Bug | 重点 P1（正确性），检查是否引入新问题 |
| 性能优化 | 重点 P2，确认优化是否引入正确性问题（P1） |
| 重构代码 | 重点 P1 + P3，确认行为无变化 |
| 上线前全量审查 | P0 → P1 → P2 → P3 全量检查 |

如果用户未指定审查范围，默认审查**当前打开的文件**以及该文件直接关联的 Service/DAO/XML 文件。
