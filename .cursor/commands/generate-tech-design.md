# generate-tech-design

## 概述
根据用户提供的需求描述，结合项目代码现状，生成一份完整的技术方案设计文档，输出到 `docs/` 目录。

## 输入要求
用户需提供以下信息：
- **需求描述**：一句话或一段话说明要做什么（如"新增用户反馈功能"、"接入第三方支付"）
- **需求文档**（可选）：PRD 链接或粘贴的需求详情
- **涉及的现有接口或模块**（可选）：帮助定位改动范围

如果用户只给了一句话需求，先进入"需求澄清"环节再开始设计。

---

## 执行步骤

### 第一步：需求澄清

分析用户输入，判断信息是否充分。如果存在以下模糊点，**先提问确认再继续**：

- 功能边界不清楚（做到什么程度？）
- 涉及的用户角色不明确（谁用？什么场景？）
- 与现有功能的关系不清楚（新增还是改造？）
- 数据来源/去向不明确
- 是否涉及异步处理、MQ、定时任务
- 是否涉及第三方服务调用

澄清完毕后，将需求总结为一段清晰的描述，供用户确认。

### 第二步：代码现状分析

在项目中搜索与需求相关的代码，了解现有架构：

1. **搜索相关 Controller**：确认是否有可复用的接口入口
2. **搜索相关 Service**：确认是否有相似的业务逻辑可参考
3. **搜索相关 DAO / Mapper**：确认是否有可复用的数据访问
4. **搜索相关 Model**：确认是否有已存在的实体类、VO、DTO
5. **查看 Redis 使用模式**：参考 `.cursor/rules/redis-usage.mdc` 中的规范
6. **查看 MyBatis 规范**：参考 `.cursor/rules/mybatis-xml.mdc` 中的规范

分析结果记录为：
- 可复用的现有代码
- 需要新增的代码
- 需要修改的代码

### 第三步：设计技术方案

读取 `docs/技术方案模板.md` 作为文档结构框架，按以下要求填充每个章节：

#### 章节一：需求概述
- 背景和目标直接从需求描述中提炼
- 功能范围要明确"包含"和"不含"

#### 章节二：业务流程
- **必须画 Mermaid 流程图**，从用户发起请求到最终返回
- 所有分支判断（校验、权限、限制）都要体现
- 如果涉及多系统交互，补充时序图
- 列出所有业务规则

#### 章节三：接口设计
- 每个接口独立一节
- 严格定义入参（字段、类型、必填、校验规则）
- 响应使用 `CommonResp<T>` 包装
- 列出所有可能的错误码（参考项目中 `ErrorCode` 常量）

#### 章节四：数据模型设计
- 新增表要给出完整的字段定义 + 索引 + DDL
- 变更表要明确写出 ALTER 语句
- 字段命名遵循项目现有的 snake_case 风格

#### 章节五：技术方案
- 核心实现思路要具体到"用什么方式解决什么问题"
- 如有多种方案，用对比表格分析优劣并给出推荐
- 涉及 Redis 的写出 Key 设计和过期策略
- 涉及 MQ 的写出 Topic/Tag 和消息体结构
- 涉及并发的写出锁策略（Redisson 分布式锁 key 设计）

#### 章节六：改动清单
- **必须精确到文件级别**
- 按模块分层列出（znzmo-model → znzmo-common → znzmo-ai / znzmo-ai-mq-consumer）
- 修改文件要说明改动点和影响范围
- 遵循项目分层规范：
  - `znzmo-model`：Entity / VO / DTO
  - `znzmo-common`：DAO / Mapper XML / 公共 Service
  - `znzmo-ai`：Controller + 业务 Service
  - `znzmo-ai-mq-consumer`：MQ 消费端

#### 章节七：异常与边界处理
- 逐一列出可能出错的场景及处理方式
- 每种异常都要有对应的错误码和用户提示

#### 章节八：性能与容量评估
- 基于需求的使用频率估算调用量
- 如果数据量大，考虑分页、索引、分表策略

#### 章节九：测试要点
- 覆盖正常流程和所有异常分支
- 列出需要重点关注的测试场景

#### 章节十：上线计划
- 按依赖关系排列上线步骤（DDL 先于代码部署）
- 给出回滚方案

#### 章节十一：排期
- 按阶段拆分工时估算

### 第四步：输出文档

文件命名规则：`[需求名称]技术方案.md`，放在 `docs/` 目录下。

输出完成后，向用户展示文档摘要：
```
## ✅ 技术方案已生成

**文件**：docs/[需求名称]技术方案.md

### 方案摘要
- **接口数量**：x 个
- **新增表**：x 张
- **改动文件**：x 个（新增 x + 修改 x）
- **核心技术点**：[一句话概括]
- **预估工时**：x 天

### 建议下一步
1. 组织技术方案评审
2. 评审通过后执行 `/create-feature-branch` 创建开发分支
3. 按改动清单逐步开发
```

---

## 质量要求

- **完整性**：模板中每一个章节都要填写，没有内容的标注"本次不涉及"
- **准确性**：改动清单中的模块和文件路径必须与项目实际结构一致
- **可操作性**：开发者拿到文档后能直接开始编码，不需要再猜
- **一致性**：接口风格、命名规范、错误码风格与项目现有代码保持统一

## 注意事项

- 依赖注入统一使用 `@Resource`，不用 `@Autowired`
- 返回值统一使用 `CommonResp<T>` 封装
- 日志统一使用 `@Slf4j`
- 所有回复和文档内容使用中文
- 如果需求过于复杂（预估超过 5 个接口），建议拆分为多期，先出第一期方案
