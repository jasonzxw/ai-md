# generate-api-doc

## 概述
针对用户指定的 API 接口，深度追踪全部业务处理链路，生成包含流程图和分步说明的完整业务流程文档，输出到 `docs/` 目录。

## 输入要求
用户需提供以下信息（至少一项）：
- 接口所在 Controller 文件路径 + 方法名或行号范围
- 接口的 URL 路径（如 `/aiDraw/aiPainting`）

如果用户未明确指定，则分析**当前打开文件中光标所在的接口方法**。

---

## 执行步骤

### 第一步：定位接口入口
1. 读取目标 Controller 文件，找到接口方法的完整代码
2. 记录：HTTP 方法、URL 路径、请求参数类型、返回值类型
3. 确认接口方法体中调用的核心 Service 方法

### 第二步：追踪 Service 层主方法
1. 找到 Service 接口定义和 Impl 实现类
2. **完整阅读**主方法的全部代码（不可跳过任何分支）
3. 梳理主方法的整体骨架：前置校验 → 核心处理 → 数据持久化 → 异步/MQ 发送 → 返回

### 第三步：递归追踪子方法
对主方法中调用的每一个**业务子方法**，记录：
- 方法名与所在文件的完整路径
- 方法所在行号范围
- 核心功能（一两句话）
- 调用的外部服务或 DAO

**追踪深度规则：**
- 直接业务子方法：必须追踪（如校验、扣费、记录创建等）
- DAO 方法：记录方法名即可，不追踪到 XML
- 工具方法（如 `StringUtils.isEmpty`）：跳过
- 外部服务调用（HTTP/MQ）：记录目标地址和用途

### 第四步：追踪异步链路（如有）
如果主方法中存在 MQ 发送或异步调用：
1. 找到 MQ Topic/Tag 的常量定义
2. 找到对应的 Consumer/Listener
3. 追踪消费端的核心处理方法（至少到第一层 dispatchTask 级别）

### 第五步：追踪请求/响应模型
1. 找到请求参数类（如 `XxxRequest`），记录文件路径和关键字段
2. 找到核心数据实体类（如数据库记录对象），记录文件路径和关键字段

### 第六步：生成文档

文档文件命名规则：`{接口方法名}接口业务流程文档.md`，放在 `docs/` 目录下。

文档必须包含以下 **五个章节**：

---

#### 章节一：整体流程图

使用 Mermaid `flowchart TD` 格式，要求：
- 从客户端请求开始，到最终返回结束
- 每个关键步骤用中文标注
- 所有分支判断和错误返回都要体现
- 异步链路（MQ消费端）也要包含在图中
- 步骤编号标注（①②③...），与后续章节对应

#### 章节二：分阶段详细说明

按处理阶段分组（如 Controller 层、前置校验、核心处理、异步执行等），每个阶段用表格描述：

| 序号 | 功能 | 方法 | 行号 | 核心逻辑 |
|------|------|------|------|----------|

- **方法**列：写方法名，如果跨文件需标注简要路径
- **行号**列：写方法定义的起止行号
- **核心逻辑**列：一两句话说清该步骤做了什么，调用了哪些外部依赖

#### 章节三：关键数据流

用文本流程图展示数据对象的转换关系：
```
请求对象 → 业务记录对象 → 关联记录 → MQ消息 → 异步处理
```

#### 章节四：核心文件索引

| 模块 | 文件 | 关键方法/类 |
|------|------|------------|

列出接口涉及的所有关键文件，每个文件标注其中最重要的方法。

#### 章节五：错误码汇总

| 错误码 | 含义 | 触发条件 |
|--------|------|----------|

列出接口所有可能返回的错误码及其触发场景。

---

## 质量要求

- **完整性**：不遗漏任何业务分支，包括特殊渠道、AB 实验、异步链路
- **精确性**：每个方法都标注准确的文件路径和行号范围
- **可读性**：流程图清晰、表格对齐、层次分明
- **实用性**：新人阅读此文档后，能快速理解接口的完整业务逻辑

## 注意事项

- 如果接口逻辑非常庞大（主方法超过 200 行），优先保证流程图和表格的完整性
- 对于条件分支极多的方法（如引擎类型决策），可适当归纳而非逐一列出
- 追踪子方法时使用并行搜索提高效率
- 文档中的行号以分析时的代码为准，标注"行号可能因代码变更而偏移"
