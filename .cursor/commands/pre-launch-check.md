# pre-launch-check

## 概述
上线前综合检查。自动扫描当前分支的所有改动，逐项核查代码、数据库、配置、缓存、MQ 等维度，生成上线检查报告，确保不带问题上线。

## 执行步骤

### 第一步：采集改动范围
```bash
git log master..HEAD --oneline
git diff master..HEAD --name-only
git diff master..HEAD --stat
```

从输出中提取：
- 本次上线包含的**全部提交**（逐条列出）
- 涉及变更的**全部文件**
- 按模块分类统计（znzmo-model / znzmo-common / znzmo-ai / znzmo-ai-mq-consumer / znzmo-ai-api）

如果无法对比 master（如 master 不存在），改用远程分支：`git diff origin/master..HEAD`

### 第二步：编译校验
```bash
mvn clean compile -q
```
- 编译成功：记录 ✅
- 编译失败：记录 ❌，输出错误详情，**标记为阻断项**

### 第三步：逐维度检查

读取所有变更文件的内容，按以下 7 个维度逐项检查：

---

#### 维度一：代码质量

| 检查项 | 说明 |
|--------|------|
| 新增接口是否有鉴权 | 检查 Controller 中新增的 `@RequestMapping`/`@PostMapping` 方法是否获取了 accountId 并做了判空 |
| 是否存在 `e.printStackTrace()` | 全部改用 `log.error()` |
| 是否有未处理的空指针风险 | DAO 查询结果、链式调用、集合操作前的判空 |
| 异常是否被正确捕获 | catch 块中是否记录了日志，是否有吞异常的情况 |
| 返回值是否使用 CommonResp | 新增接口是否遵循统一响应格式 |
| 依赖注入是否使用 @Resource | 不允许使用 @Autowired |

---

#### 维度二：数据库变更

| 检查项 | 说明 |
|--------|------|
| 是否有新增表 | 检查是否有新的 Entity 类或 Mapper XML 中的 CREATE 语句 |
| 是否有变更字段 | 检查 Mapper XML 中是否有新增的列引用，对应 Entity 中是否有新字段 |
| DDL 是否已准备 | 如果有表结构变更，提醒确认 DDL 脚本是否已准备并在预发环境验证 |
| 新增 SQL 是否有索引支撑 | 检查 WHERE 条件中的字段是否可能需要索引 |
| 是否存在 ${} 拼接 | MyBatis XML 中的 SQL 注入风险检查 |

---

#### 维度三：缓存变更（Redis）

| 检查项 | 说明 |
|--------|------|
| 是否有新增 Redis Key | 搜索变更文件中的 `RedisConstKey`、`redisTemplate`、`redissonClient` 等关键词 |
| Key 命名是否规范 | 是否遵循 `模块:业务:标识` 格式 |
| 是否设置了过期时间 | 新增的缓存是否有 TTL，避免内存泄漏 |
| 数据变更时缓存是否同步 | 写库操作后是否清理/更新了对应缓存 |

---

#### 维度四：MQ 变更

| 检查项 | 说明 |
|--------|------|
| 是否有新增 Topic/Tag | 搜索变更文件中的 `TOPIC_`、`TAG_`、`sendMessage` 等关键词 |
| Consumer 是否已实现 | 如果新增了 MQ 发送，确认消费端是否已在 znzmo-ai-mq-consumer 中实现 |
| 消息幂等性 | 消费端是否有防重处理（如 Redisson 锁） |
| 消息发送失败处理 | 发送失败时是否有补偿机制或状态回退 |

---

#### 维度五：配置变更

| 检查项 | 说明 |
|--------|------|
| 是否修改了配置文件 | 检查 `application.yml`、`application-*.yml` 是否有变更 |
| 新增配置项是否各环境一致 | 提醒确认 dev/test/prod 环境的配置是否已同步 |
| 是否有硬编码的配置值 | URL、超时时间、阈值等是否应该抽到配置文件 |

---

#### 维度六：兼容性

| 检查项 | 说明 |
|--------|------|
| 接口入参是否新增了必填字段 | 如果旧版客户端还在调用，新增必填字段会导致旧版报错 |
| 接口返回值是否删除了字段 | 删除字段可能导致前端解析异常 |
| 数据库字段是否有 NOT NULL 且无默认值 | 新增 NOT NULL 字段如果历史数据没填充会导致写入失败 |
| 部署顺序是否有依赖 | 是否需要先执行 DDL 再部署代码，或者先部署消费者再部署生产者 |

---

#### 维度七：回滚方案

| 检查项 | 说明 |
|--------|------|
| 代码是否可直接 revert | 评估 `git revert` 回滚是否会影响其他功能 |
| 数据库变更是否可逆 | 新增表/字段可以保留；如果有数据迁移，是否有回滚脚本 |
| 是否有功能开关 | 关键功能是否可通过配置开关快速关闭，而无需回滚代码 |

---

### 第四步：生成检查报告

按以下格式输出：

```
## 🚀 上线前检查报告

**分支**：xxx
**提交数**：x 个
**变更文件数**：x 个
**编译状态**：✅ 通过 / ❌ 失败

---

|------|------|--------|

```

### 第五步：给出结论

根据检查结果给出最终结论：

- **可以上线** ✅：无阻断项，需确认项已全部标注
- **建议修复后上线** ⚠️：有需确认项但无阻断项，列出建议修复的内容
- **不建议上线** ❌：存在阻断项，必须修复后重新检查

## 注意事项
- 检查范围严格限定在 `master..HEAD` 的差异文件，不检查未改动的代码
- 对于"需确认项"，只提出问题和建议，不要替用户做决定
- 如果某个维度完全不涉及（如没有 Redis 变更），标记为 ✅ 并注明"本次不涉及"
- 报告中不要包含笼统的建议，每条发现项都要具体到文件和代码位置
