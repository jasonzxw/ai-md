# check-redis-usage

## 概述
检查当前打开的文件（或用户指定的文件）中的 Redis 操作代码，确保符合项目规范，排查潜在风险。

## 执行步骤

### 第一步：扫描 Redis 操作
在目标文件中查找所有 Redis 相关操作：
- `redisTemplate` / `stringRedisTemplate` 的调用
- `RedisConstKey` 或直接拼接的 Key 字符串
- `opsForValue` / `opsForHash` / `opsForList` / `opsForSet` / `opsForZSet` 操作
- `@Cacheable` / `@CacheEvict` / `@CachePut` 注解

### 第二步：逐项检查
对每一处 Redis 操作，检查以下内容：

#### Key 规范
- [ ] Key 是否使用 `RedisConstKey` 中定义的常量？如果是硬编码字符串，标记为 `[Should Fix]`
- [ ] Key 命名是否有业务语义、是否包含必要的隔离前缀？
- [ ] Key 是否存在与其他业务冲突的风险？

#### 过期时间
- [ ] 是否设置了过期时间（`expire` / `timeout` 参数）？
- [ ] 永不过期的 Key 是否有明确理由？如无，标记为 `[Should Fix]`
- [ ] 过期时间是否合理（不要过短导致缓存命中率低，不要过长导致数据不一致）？

#### 缓存一致性
- [ ] 数据变更（增/删/改）时，是否同步删除或更新了对应缓存？
- [ ] 是否存在"先删缓存再更新 DB"导致的并发不一致？建议"先更新 DB 再删缓存"
- [ ] 缓存读取为空时，是否有回源查 DB 的逻辑？

#### 异常与降级
- [ ] Redis 操作是否有 try-catch？失败时是否回退到 DB 查询而非直接抛异常中断主流程？
- [ ] 是否存在缓存击穿风险（热点 Key 过期瞬间大量请求打到 DB）？是否需要加互斥锁？
- [ ] 是否存在缓存穿透风险（查询不存在的数据反复穿透到 DB）？是否需要缓存空值？

#### 大 Key 风险
- [ ] 是否存在将大量数据（整个列表、大 Map）直接缓存的情况？
- [ ] Value 序列化后预估大小是否超过 10KB？如超过，建议拆分或压缩

### 第三步：输出检查报告
按以下格式输出：

```
## Redis 使用检查报告

### 检查范围
- 文件：`xxx.java`
- Redis 操作数量：x 处

### 发现问题
#### [严重等级] 问题简述
- **位置**：`文件:行号`
- **当前代码**：`相关代码片段`
- **问题**：具体说明
- **建议**：修复方案

### 总结
- 问题总数：x
- Must Fix：x | Should Fix：x | Suggestion：x
```
